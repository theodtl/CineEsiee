1. Idee generale du projet

Le programme simule une petite base de donnees cinematographique. Il gere des acteurs, des realisateurs, des films et un petit graphe de cinemas pour calculer un trajet optimal.

Les donnees ne sont pas codees en dur dans main.c. Elles sont lues au demarrage depuis donnees.csv, puis stockees en memoire dans des structures et des listes chainees.

Structures Listes chainees Pointeurs CSV Dijkstra 0 warning gcc
Phrase a sortir a l'oral : "CineEsiee est une base de donnees simulee en C. Les donnees viennent d'un CSV, puis le programme les relie en memoire avec des pointeurs pour eviter les duplications."

2. Organisation des fichiers

main.c

Contient la fonction main, le menu console, les saisies utilisateur et les appels vers les autres modules.

personne.h / personne.c

Definit et manipule les types Date, Personne, Acteur, Realisateur, ainsi que les listes chainees de personnes.

film.h / film.c

Definit les films, les genres, l'affichage, la recherche et les filmographies.

cinema.h / cinema.c

Definit le graphe de cinemas, les routes entre cinemas et l'algorithme de Dijkstra.

fichier.h / fichier.c

Charge les donnees depuis donnees.csv et construit les listes en memoire.

donnees.csv

Contient toutes les donnees : acteurs, realisateurs, films, cinemas et routes.

3. Relations entre les fichiers

main.c inclut les modules dont il a besoin :

#include "cinema.h"
#include "fichier.h"
#include "film.h"
#include "personne.h"

film.h inclut personne.h, car un film contient des pointeurs vers un Realisateur et des Acteur.

fichier.h inclut cinema.h, film.h et personne.h, car le chargement CSV remplit toute la base.

cinema.h est independant des films et des personnes : il gere uniquement le graphe.

main.c |-- fichier.h -> charge donnees.csv |-- film.h -> films + filmographies |-- personne.h -> acteurs + realisateurs |-- cinema.h -> graphe + Dijkstra film.h |-- personne.h -> Film contient Acteur* et Realisateur*

4. Structures principales

Date

typedef struct
{
    int jour;
    int mois;
    int annee;
} Date;

Elle sert a stocker une date de naissance.

Personne

typedef struct
{
    char nom[TEXTE_COURT];
    char prenom[TEXTE_COURT];
    char nationalite[TEXTE_COURT];
    Date naissance;
} Personne;

Personne contient les informations communes aux acteurs et aux realisateurs : nom, prenom, date de naissance et nationalite.

Acteur et Realisateur

typedef struct
{
    Personne infos;
} Acteur;

typedef struct
{
    Personne infos;
} Realisateur;

Ils sont deux types differents, mais ils reutilisent la structure Personne.

Listes chainees d'acteurs et de realisateurs

typedef struct NoeudActeur
{
    Acteur acteur;
    struct NoeudActeur *suivant;
} NoeudActeur;

typedef struct NoeudRealisateur
{
    Realisateur realisateur;
    struct NoeudRealisateur *suivant;
} NoeudRealisateur;

Chaque noeud contient une donnee et un pointeur vers le noeud suivant. Cela permet d'avoir un nombre variable d'acteurs ou de realisateurs.

Genre

typedef enum
{
    ACTION = 1,
    HORREUR,
    COMEDIE,
    DOCUMENTAIRE,
    POLICIER,
    DRAME,
    ANIMATION,
    SCIENCE_FICTION
} Genre;

L'enum limite les genres aux valeurs prevues par le sujet.

Film

typedef struct Film
{
    char titre[TITRE_MAX];
    int annee;
    int duree;
    Genre genre;
    Realisateur *realisateur;
    Acteur *acteurs[NB_ACTEURS_MAX];
    int nb_acteurs;
    struct Film *suivant;
} Film;

Un film contient son titre, son annee, sa duree, son genre, un pointeur vers son realisateur, un tableau de pointeurs vers ses acteurs et un pointeur vers le film suivant.

Point essentiel : le film ne copie pas les acteurs et le realisateur. Il garde seulement leurs adresses.

Cinema, Arete et GrapheCinema

typedef struct Arete
{
    Cinema *destination;
    int distance;
    struct Arete *suivant;
} Arete;

struct Cinema
{
    int id;
    char nom[CINEMA_NOM_MAX];
    Arete *voisins;
    Cinema *suivant;
};

typedef struct
{
    Cinema *cinemas;
} GrapheCinema;

Le graphe contient une liste chainee de cinemas. Chaque cinema contient une liste chainee de routes vers ses voisins.

5. Chargement depuis donnees.csv

Le CSV contient plusieurs types de lignes. Le premier champ indique le type de donnee.

ACTEUR;nom;prenom;jour;mois;annee;nationalite
REALISATEUR;nom;prenom;jour;mois;annee;nationalite
FILM;titre;annee;duree;genre;nom_realisateur;nom_acteur1;nom_acteur2;nom_acteur3
CINEMA;id;nom
ROUTE;id_depart;id_arrivee;distance
1
Acteurs
Ils sont charges en premier pour pouvoir etre references par les films.
2
Realisateurs
Ils sont charges avant les films pour que chaque film pointe vers un realisateur existant.
3
Films
Chaque film cherche son realisateur et ses acteurs par nom, puis stocke leurs pointeurs.
4
Cinemas
Les sommets du graphe sont crees.
5
Routes
Les routes relient des cinemas deja crees.
Limite du CSV : les champs ne doivent pas contenir le caractere ;. Le format est volontairement simple pour rester defendable dans un TP C.

6. Deroulement du programme

  1. main declare les listes : acteurs, realisateurs, films et graphe.
  2. initialiser_graphe(&graphe) met la liste de cinemas a NULL.
  3. charger_base("donnees.csv", ...) charge toutes les donnees.
  4. Le programme entre dans une boucle while (choix != 4).
  5. afficher_menu affiche le menu.
  6. lire_choix lit le choix utilisateur.
  7. Le switch execute la fonction correspondante.
  8. A la fin, le programme libere la memoire.
liberer_films(films);
liberer_acteurs(acteurs);
liberer_realisateurs(realisateurs);
liberer_graphe(&graphe);

Dijkstra dans le projet

optimiser_trajet utilise quatre tableaux :

  • tableau : contient les pointeurs vers les cinemas ;
  • distance : meilleure distance connue depuis le depart ;
  • visite : indique si un cinema est deja traite ;
  • precedent : permet de reconstruire le chemin final.
1
Toutes les distances valent INF, sauf le depart qui vaut 0.
2
On choisit le cinema non visite qui a la plus petite distance.
3
On regarde ses voisins et on met a jour leurs distances si un chemin plus court est trouve.
4
A la fin, on reconstruit le chemin en remontant le tableau precedent.
Lien avec OSPF : OSPF utilise une logique de plus court chemin. Ici, Dijkstra calcule le meilleur trajet entre deux cinemas.

12. Pourquoi les donnees ne sont pas dupliquees ?

Dans un mauvais modele, chaque film pourrait contenir directement toutes les informations de son realisateur : nom, prenom, date de naissance et nationalite. Si David Fincher realise trois films, ces informations seraient copiees trois fois.

Dans ce projet, un film contient seulement :

Realisateur *realisateur;
Acteur *acteurs[NB_ACTEURS_MAX];

Les personnes sont stockees une seule fois dans leurs listes. Les films pointent vers ces personnes.

Liste realisateurs [David Fincher] <---- Fight Club ^ <---- Seven | <---- The Game Liste acteurs [Brad Pitt] <---- Fight Club ^ <---- Seven

13. Pourquoi utiliser des listes chainees ?

Le sujet indique qu'une liste lineaire chainee est judicieuse pour optimiser l'allocation memoire.

Avantages

  • Pas besoin de connaitre le nombre d'elements a l'avance.
  • Allocation dynamique avec malloc.
  • On alloue seulement ce dont on a besoin.

Limite

La recherche est lineaire : il faut parcourir la liste depuis le debut.

14. Pourquoi separer .h et .c ?

Les fichiers .h contiennent les structures, constantes et prototypes. Les fichiers .c contiennent l'implementation des fonctions.

Cette separation rend le projet plus clair : chaque module a une responsabilite precise et main.c reste lisible.

A dire au prof : "Les .h decrivent l'interface du module, les .c contiennent le code reel."

15. Questions possibles et reponses

1. Est-ce que votre programme est une vraie base de donnees ?

Non, ce n'est pas une base SQL. C'est une base de donnees simulee en C. Les donnees sont stockees dans des structures et listes chainees en memoire, et chargees depuis un CSV.

2. Ou sont stockees les donnees ?

Sur disque, elles sont dans donnees.csv. En memoire, elles sont dans NoeudActeur *acteurs, NoeudRealisateur *realisateurs, Film *films et GrapheCinema graphe.

3. Comment evitez-vous la duplication des donnees ?

Un film ne copie pas les informations des acteurs et du realisateur. Il contient des pointeurs vers des acteurs et realisateurs deja stockes dans leurs listes.

4. Pourquoi la filmographie n'est-elle pas stockee dans l'acteur ?

Pour eviter la redondance. La filmographie est calculee en parcourant la liste des films et en verifiant si l'acteur apparait dans chaque film.

5. Que se passe-t-il quand on cherche la filmographie de David Fincher ?

Le programme cherche le realisateur Fincher, parcourt tous les films et affiche ceux dont film->realisateur pointe vers ce realisateur.

6. Pourquoi comparer des pointeurs au lieu de comparer des noms ?

Parce que les films referencent directement les personnes existantes. Deux pointeurs egaux designent exactement la meme personne en memoire.

7. Pourquoi les acteurs et realisateurs sont avant les films dans le CSV ?

Parce qu'un film doit pointer vers des acteurs et un realisateur deja crees. Si on lit les films avant, les pointeurs n'existent pas encore.

8. Pourquoi utiliser malloc ?

malloc permet d'allouer dynamiquement les noeuds des listes chainees et les elements du graphe selon le contenu du CSV.

9. Ou liberez-vous la memoire ?

A la fin de main, avec liberer_films, liberer_acteurs, liberer_realisateurs et liberer_graphe.

10. Pourquoi liberer les films ne libere pas les acteurs ?

Les films ne possedent pas les acteurs. Ils ont seulement des pointeurs vers eux. Les acteurs sont liberes par leur propre liste.

11. Pourquoi utiliser un enum pour les genres ?

Un enum limite les genres aux valeurs imposees par le sujet et evite les chaines incoherentes.

12. Comment fonctionne la recherche d'un film ?

Le programme parcourt la liste chainee des films et compare le titre saisi avec le titre du film. La comparaison ignore les majuscules/minuscules.

13. Pourquoi utiliser fgets plutot que scanf ?

fgets lit une ligne complete et evite les problemes de retour a la ligne restant dans le buffer clavier. Ensuite sscanf convertit si besoin.

14. Comment fonctionne le chargement CSV ?

charger_base ouvre le fichier avec fopen, lit les lignes avec fgets, les decoupe avec strtok, puis cree les bonnes structures selon le premier champ.

15. Quelles sont les limites du CSV ?

Le format est volontairement simple : les champs ne doivent pas contenir ;, et l'ordre des donnees doit etre respecte.

16. Comment est represente le graphe ?

Le graphe contient une liste de cinemas. Chaque cinema contient une liste de routes vers ses voisins.

17. Pourquoi ajouter une route dans les deux sens ?

Parce que les routes sont considerees comme non orientees : si on peut aller de A a B, on peut aussi aller de B a A.

18. Quel algorithme utilisez-vous pour le trajet ?

Dijkstra. C'est coherent avec OSPF, qui utilise aussi une logique de plus court chemin.

19. Expliquez Dijkstra simplement.

On part du cinema de depart avec une distance 0. A chaque etape, on choisit le cinema non visite avec la plus petite distance, puis on met a jour les distances de ses voisins.

20. A quoi sert le tableau precedent ?

Il permet de reconstruire le chemin final en remontant depuis le cinema d'arrivee vers le depart.

21. Pourquoi utiliser INF ?

INF represente une distance initialement inconnue ou infinie.

22. Que se passe-t-il si un cinema n'existe pas ?

optimiser_trajet affiche un message d'erreur et s'arrete proprement.

23. Pourquoi certaines fonctions sont static ?

static limite la fonction au fichier .c. C'est utile pour les fonctions internes comme copier ou decouper_ligne.

24. Pourquoi utiliser des fichiers .h ?

Les .h contiennent les prototypes et structures visibles par les autres fichiers. Ils separent l'interface de l'implementation.

25. Comment compiler le projet ? 
gcc -Wall main.c personne.c film.c cinema.c fichier.c -o CineEsiee.exe

Version stricte :

gcc -Wall -Wextra -pedantic -std=c99 main.c personne.c film.c cinema.c fichier.c -o CineEsiee.exe
26. Pourquoi viser 0 warning ?

Les warnings signalent souvent des erreurs potentielles. Un programme a 0 warning est plus propre et plus fiable.

27. Que se passe-t-il si donnees.csv est absent ?

charger_base retourne 0, puis main affiche une erreur et quitte avec return 1.

28. Pourquoi le menu contient plus que les 4 options obligatoires ?

Les 4 options obligatoires sont presentes. Les autres sont des extensions utiles : acteurs, trajet et filmographies.

29. Quel est le role de nb_acteurs ?

nb_acteurs indique combien de cases du tableau acteurs sont reellement utilisees.

30. Pourquoi NB_ACTEURS_MAX vaut 3 ?

Le sujet demande de stocker les acteurs principaux de 1 a 3. Le tableau a donc une taille maximale de 3.

16. Resume oral en 1 minute

"Mon projet CineEsiee simule une base de donnees de films en C. Les donnees sont lues depuis donnees.csv. En memoire, j'utilise des listes chainees pour les acteurs, les realisateurs et les films. Un film contient un pointeur vers son realisateur et un tableau de pointeurs vers ses acteurs principaux, donc les informations des personnes ne sont pas dupliquees."

"Le menu permet de lister les films, lister les realisateurs, rechercher un film, afficher des filmographies et calculer un trajet entre cinemas. La partie trajet utilise un graphe pondere et l'algorithme de Dijkstra, inspire du fonctionnement d'OSPF. A la fin du programme, toute la memoire allouee dynamiquement est liberee."

Points faibles a connaitre

  • Le CSV est simple et ne gere pas les champs contenant ;.
  • La recherche d'acteur/realisateur se fait par nom exact.
  • Le programme ne modifie pas le CSV depuis le menu.
  • Le trajet calcule le plus court chemin entre deux cinemas, pas une tournee complete avec plusieurs livraisons.
  • La recherche dans les listes chainees est lineaire.
